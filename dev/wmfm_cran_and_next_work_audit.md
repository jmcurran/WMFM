# WMFM CRAN-readiness and next-development audit

Source inspected: `stage37_6_1_completed.zip` from the current Stage 37 stream.

## Summary

WMFM looks close to a CRAN-preparation phase, but I would not submit immediately. The package should first go through a dedicated CRAN-hardening stage focused on policy compliance, examples, external-service behavior, and user-facing documentation.

The package is modest in size, has extensive tests, and appears well organised. The main CRAN risks are not package size or obvious missing files; they are that the package is a Shiny/LLM-facing package with external-provider configuration, local-server defaults, and many generated documentation/tests. CRAN will be sensitive to examples/tests that contact services, write outside temporary locations, or depend on optional credentials.

## CRAN-readiness concerns

### 1. External services and LLM provider behavior

WMFM depends on `ellmer` and supports Ollama, Claude, OpenAI, and OpenAI-compatible providers. That is acceptable in principle, but CRAN checks must never require credentials, live API access, or a running Ollama server.

Items to verify:

- All examples involving `getChatProvider()`, `runExample()`, app launch, or LLM generation should be wrapped in `\dontrun{}` or otherwise avoid external calls.
- All tests must keep using fake providers or mocks.
- Startup paths must not try to discover Ollama models or contact provider APIs during package load, examples, tests, or help-page execution.
- Any provider discovery must happen only through explicit user action in the Shiny app or explicit user function calls.

### 2. Default Ollama URL

The provider defaults include a concrete URL:

```text
http://corrin.stat.auckland.ac.nz:11434
```

For CRAN, this is risky. It may look like a hard-coded institutional service endpoint. Prefer a localhost default such as:

```text
http://localhost:11434
```

or no network default unless the user explicitly configures it. If the institutional URL is important locally, keep it in local config, not package defaults.

### 3. DESCRIPTION wording and dependency positioning

The DESCRIPTION says explanations are generated by a large language model. That is honest, but for CRAN it may help to state that external providers are user-configured and not required during package checks.

Also review whether all Imports are essential at load/runtime. In particular:

- `ellmer` is likely necessary if provider functions are core.
- `sodium` appears to support provider/developer authentication helpers.
- `devtools` is only in Suggests, which is good.
- `rstudioapi` and `clipr` are in Suggests, which is also good.

### 4. Examples and tests

There are 30 example sections. Before CRAN submission, run:

```r
rcmdcheck::rcmdcheck(args = c("--as-cran"), error_on = "warning")
```

Then inspect example timings and whether any examples attempt provider construction, app launch, local files, or network access.

### 5. Writing files and diagnostics

The package writes diagnostics JSON and provider configuration. For CRAN, writes should generally go to user-initiated paths, temporary files, or accepted user configuration directories. Verify that no example or test writes to a persistent user home directory by default.

### 6. NEWS.md and development-stage language

`NEWS.md` contains many stage and repair references. CRAN does not forbid this, but a public release NEWS file is easier to maintain if current-release entries are user-facing and development-stage archaeology is compressed.

Before CRAN, consider making a clean top section for the release version and moving detailed stage history below it.

### 7. Version number

The current version `0.3.1.010` may be technically valid as an R package version. However, for a CRAN-facing release I would consider a simpler public version, such as `0.3.1` or `0.4.0`, after the final CRAN-hardening branch is complete. Keep internal build numbers in git tags or development-stage branches rather than the CRAN release version unless you have a strong reason.

## Recommended CRAN-hardening stages

### Stage C1: CRAN preflight audit

Goal: make the package safe for `R CMD check --as-cran`.

Tasks:

- Run `devtools::check(args = "--as-cran")` or `rcmdcheck::rcmdcheck(args = c("--as-cran"))`.
- Review examples for network calls and app launches.
- Confirm all LLM-related tests use fake providers.
- Confirm no package-load side effects.
- Confirm no persistent writes during examples/tests.

### Stage C2: Provider defaults and CRAN-safe documentation

Goal: make external-provider behavior explicit and CRAN-safe.

Tasks:

- Replace institutional Ollama default with localhost or no configured default.
- Document provider configuration as opt-in.
- Ensure startup messages do not imply hidden credentials or automatic provider access.
- Add tests confirming provider discovery is not attempted without explicit user action.

### Stage C3: Public-release documentation pass

Goal: prepare README, NEWS, and help pages for external users.

Tasks:

- Add a concise README quick-start.
- Add a CRAN-safe provider-configuration section.
- Add a teaching-use section describing intended classroom workflow.
- Compress or reorganize NEWS for release clarity.
- Review exported function help pages for examples that CRAN can run safely.

### Stage C4: Package metadata polish

Goal: polish CRAN metadata.

Tasks:

- Review Title for title-case and clarity.
- Review Description for CRAN style and external-service clarity.
- Decide on public release version.
- Consider whether a package-level help file is needed.
- Confirm `LICENSE.md` and `License:` field are consistent with CRAN expectations.

## High-value next development ideas

### 1. Explanation quality polish

Continue improving generated explanations without adding major new statistical machinery.

Good targets:

- Confidence interval phrasing, especially avoiding `could plausibly lie`.
- Repetition removal.
- Better student-facing language for adjustment context, e.g. `varies across pictures` instead of `varies depending on the adjustment context`.
- More consistent concluding sentences that directly answer the research question.

### 2. Release-mode versus developer-mode boundary

Before CRAN, make sure the app has a clean release-mode experience and developer diagnostics remain hidden unless deliberately enabled.

Good targets:

- Developer-mode UI smoke tests.
- Provider settings smoke tests.
- A release-mode app startup test.
- A public-user workflow test that never exposes internal grading/scoring controls.

### 3. Example library polish

The examples are now numerous. A next useful stage would be to curate them.

Good targets:

- Separate classroom examples from internal test examples.
- Add example metadata such as difficulty, model family, and teaching topic.
- Provide a smaller public example list by default, with internal/test examples hidden or developer-only.

### 4. Scoring and grading report UX

The deterministic and LLM scoring systems have become substantial. A next development stream could make them more useful for teaching and debugging.

Good targets:

- Cleaner grade summaries.
- Better explanation of why a score was assigned.
- A compact `what to improve` section.
- Comparison reports across repeated runs.

### 5. Follow-up question UX

The deterministic follow-up machinery is mature but could be made more discoverable.

Good targets:

- Suggested follow-up prompts.
- Clearer UI when a follow-up cannot be answered deterministically.
- A small library of supported follow-up question templates.
- Better explanation of why a requested prediction is extrapolative.

## Recommended immediate next move

Create a new branch for CRAN hardening, separate from feature development. I would not mix CRAN release work with new feature work.

Suggested branch names:

- `cran-preflight`
- `cran-hardening`
- `release-prep-cran`

First local commands after creating the branch:

```bash
git status --short
R -q -e "devtools::test(stop_on_failure = TRUE, stop_on_warning = TRUE)"
R -q -e "devtools::check(args = c('--as-cran'), error_on = 'warning')"
```

Expect the `--as-cran` check to surface issues. Treat that output as the first CRAN-hardening context.
