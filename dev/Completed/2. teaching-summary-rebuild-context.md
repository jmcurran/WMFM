# Teaching summary rebuild

## Context

I am working on an R package/app called WMFM.

There is evidence that an earlier branch called `explaining-explanations` contained unmerged work that should not now be merged wholesale into the current code base. The safe interpretation is that this branch is now a reference implementation rather than a branch to merge directly.

The relevant branch history indicates that the old branch contained a teaching-summary feature, including:
- adding a student-facing explanation teaching summary
- later wording and ordering polish around that teaching summary

At the same time, the current code base has moved on substantially, including large changes to core files and the removal or reshaping of some related explanation-support code. Because of that drift, the right task is to rebuild the lost teaching-summary functionality against the current architecture rather than force a merge or blindly cherry-pick old commits.

This context is for planning and implementing that rebuild.

---

## Goal

Rebuild a student-facing teaching-summary layer for model explanations in the current WMFM code base.

This teaching summary should:
- help students understand how to read the generated explanation
- explain the role of key model components in plain language
- remain educational rather than developer-facing
- build on current explanation outputs and current app structure
- avoid resurrecting outdated branch architecture that no longer fits the code base

This stage is about recovering the lost teaching-summary capability in a clean, current-architecture-compatible way.

## Constraints and requirements

- Do not merge the old `explaining-explanations` branch wholesale.
- Do not assume the old implementation can be cherry-picked cleanly.
- Rebuild the feature so it fits the current code base structure.
- Preserve the current app and explanation architecture where possible.
- Use plain language suitable for students.
- Keep the feature deterministic where possible unless there is a strong reason not to.
- Keep code documented with roxygen2.
- Follow project style:
  - use `=` for assignment
  - use camelCase
  - always use braces
  - prefer clear helpers rather than over-fragmentation
- Add or update focused tests rather than relying on old tests unchanged.

## Suggested interpretation of the feature

The teaching summary should not merely repeat the explanation. It should act as a bridge between:
- the fitted model
- the generated plain-language explanation
- the student's understanding of what the explanation is doing

A good teaching summary would likely:
- identify the response and predictors being discussed
- explain what the explanation is mainly claiming
- clarify whether the explanation is talking about baseline levels, changes, comparisons, probabilities, odds, fitted values, or uncertainty
- help students see how the explanation answers the research question
- avoid low-level implementation details and raw internal data structures

## Rebuild strategy

Treat the old branch as a reference and rebuild in stages.

### Stage 1: inspect old implementation and recover feature intent

Tasks:
- inspect the current versions of files related to explanation output and UI
- inspect the old branch versions of:
  - `R/model-explanationTeachingSummary.R`
  - `R/app-explanationTutor.R`
  - `R/app-ui.R`
  - `R/app-server.R`
- identify:
  - what inputs the old teaching summary expected
  - what outputs it produced
  - whether it depended on any deleted claim-evidence infrastructure
  - which parts are still conceptually valuable

Deliverable:
- a concise design note describing the intended behaviour of the teaching summary in current terms

### Stage 2: define the current data contract

Tasks:
- determine what information is already available in the current `wmfmModel` object and explanation runtime
- identify what minimum inputs a rebuilt teaching summary needs
- decide whether the summary should be built from:
  - explanation text
  - explanation audit information
  - model metadata
  - a combination of these
- define a deterministic output contract for the teaching summary object or text builder

Deliverable:
- a simple, explicit contract for a rebuilt teaching-summary generator

### Stage 3: implement core teaching-summary builder

Tasks:
- implement or rewrite the core builder in `R/model-explanationTeachingSummary.R`
- keep the logic small, explicit, and testable
- avoid dragging in deleted old systems unless genuinely necessary
- separate:
  - data extraction
  - summary construction
  - final text rendering

Deliverable:
- a deterministic teaching-summary builder compatible with the current model/explanation flow

### Stage 4: implement rendering helpers

Tasks:
- create or rebuild rendering helpers for the app UI
- ensure the UI helpers can display the teaching summary clearly and simply
- confirm the rendered text is appropriate for students and not overly technical
- avoid duplicating logic between model code and UI code

Deliverable:
- UI helper functions for the rebuilt teaching summary

### Stage 5: integrate into the app

Tasks:
- wire the teaching summary into the current app flow
- determine where it should appear in the model explanation surface
- ensure it behaves sensibly when explanation data is missing or incomplete
- keep fallback behaviour graceful

Deliverable:
- current app integration for the teaching summary

### Stage 6: testing and polish

Tasks:
- add unit tests for the teaching-summary builder
- add app-level tests for visibility and rendering where practical
- verify behaviour across several model families or common examples
- polish wording only after functionality is stable

Deliverable:
- passing tests and stable student-facing wording

## Risks and conflict points

- the old teaching summary may depend on deleted explanation-claim infrastructure

- the teaching summary should probably consume current explanation-oriented information, not recreate a second competing explanation-analysis system from scratch
