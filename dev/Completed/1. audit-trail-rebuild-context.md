# Audit trail rebuild context for WMFM

## Title
Rebuild of lost explanation-audit functionality

## Context
I am working on the WMFM R package/app.

We investigated whether an older branch called `explaining-explanations` had been merged into the current development line. The Git diagnostics showed that the branch contains unique commits that are not present in `exp-to-exp-iv` and therefore should be treated as genuinely missing branch history, not duplicated work.

However, the branch also diverged heavily from the current codebase. The old branch and the current branch differ across many core files, including explanation, audit, app UI/server, prompt, CI-data, and documentation files. This means the correct strategy is not to merge the old branch wholesale, but to recover the underlying ideas and rebuild them safely against the current architecture.

This context is for the first recovery stream only:

**the explanation-audit trail**

The audit trail is intended to provide a deterministic, inspectable record of how the model explanation was constructed, so that later student-facing or UI-facing explanation-support features can build on it.

## Confirmed Git evidence
The following commits on `origin/explaining-explanations` were shown as unique relative to `exp-to-exp-iv` via `git cherry -v`:

- `1b67646` Add explanation-construction audit trail for model narratives
- `5f0ce8d` Add student-facing explanation teaching summary
- `742498a` Polish teaching-summary wording and variable chips
- `b48e297` Polish teaching summary wording and ordering
- `b4ed159` Create dedicated Model Explanation tab and guided tutor helper
- `120891b` Fix escaped regexes in model explanation UI tests
- `bdcd8f1` added `.wmfm_built_pkg_path` to `.Rbuildignore`

So the audit-trail commit appears to represent genuinely unmerged history.

## Important architectural conclusion
Do **not** try to recover this by merging the whole old branch.

Instead:

- treat `explaining-explanations` as a reference implementation
- recover only the intended audit functionality
- rebuild it against the current architecture on `exp-to-exp-iv`
- avoid resurrecting outdated or deleted subsystems unless they are still clearly needed

## Current working assumption
The current codebase may already contain some audit-related files, tests, or partial logic. Therefore this rebuild should begin with a **verification and gap-analysis stage**, not with blind reimplementation.

That means the first task is to determine:

1. what audit functionality already exists in the current codebase
2. what was intended in the lost branch
3. what gaps still remain relative to the intended audit design

## Goal of this rebuild
Re-establish a deterministic explanation-audit subsystem that:

- is built during model/explanation generation
- is attached to the returned `wmfmModel` object in a stable way
- records the key structured ingredients used in explanation construction
- is testable without requiring live LLM access
- can later support teaching summaries and a student-facing explanation panel

## Non-goals for this stage
This stage should **not** yet attempt to fully rebuild:

- the student-facing teaching summary
- the dedicated Model Explanation tab
- the guided tutor helper
- broad claim-evidence UI features unless they are strictly required for the audit foundation

Those belong to later rebuild streams.

## Rebuild philosophy
The rebuild should prefer:

- deterministic R-side data structures
- stable object attachment to `wmfmModel`
- minimal coupling to transient UI logic
- explicit tests over prompt-only expectations
- backward-compatible extension where sensible

The rebuild should avoid:

- reintroducing stale architecture from the old branch wholesale
- coupling audit generation too tightly to Shiny app state
- hiding critical explanation inputs only inside prompts
- depending on live LLM calls for audit correctness

## Proposed staged schedule of work

### Stage 1. Audit inventory and gap analysis
Purpose: establish what is already present and what is missing.

Tasks:
- inspect current files related to audit generation, storage, printing, and runtime use
- inspect old branch files and commit intent for the audit-trail feature
- list the intended audit fields and sections
- compare current implementation against intended behaviour
- decide whether the current codebase already has:
  - a complete audit object
  - a partial audit object
  - stale or mismatched audit logic
- produce a short gap report before editing code

Likely files to inspect:
- `R/api-runModel.R`
- `R/class-wmfmModel.R`
- `R/model-explanationAudit.R`
- `R/methods-print-wmfmExplanationAudit.R`
- `R/app-equation-runtime.R`
- `R/app-explanationAudit-uiHelpers.R`
- `tests/testthat/test-buildModelExplanationAudit.R`
- `tests/testthat/test-explanation-audit-runtime.R`

Deliverable:
- a precise inventory of what exists now and what still needs rebuilding

### Stage 2. Define the audit contract
Purpose: lock down what the deterministic audit object must contain.

Tasks:
- define the required audit sections and field names
- clarify which values must be deterministic and precomputed in R
- clarify which values are explanatory metadata versus raw model quantities
- decide the expected class/shape of the audit object
- document invariants such as:
  - must be present after `runModel()`
  - must not require an LLM call
  - must be printable and inspectable

Possible audit sections to confirm:
- overview / model metadata
- prompt inputs or explanation inputs
- interpretation scale information
- numeric anchor / reference-value information
- reference levels
- coefficient table
- confidence interval information
- baseline evidence and effect evidence
- any additional structured explanation-support inputs that are still justified

Deliverable:
- a written audit contract that can drive implementation and tests

### Stage 3. Rebuild or repair audit construction logic
Purpose: ensure the audit object is built deterministically from the current modelling pipeline.

Tasks:
- refactor or rebuild `buildModelExplanationAudit()` and helpers as needed
- make sure all audit quantities are derived from the current code path, not stale assumptions from the old branch
- ensure the audit logic uses current helper functions for CI, numeric anchors, response scales, reference levels, and fitted values
- keep helper boundaries clean and testable

Implementation preferences:
- small helper functions where they improve clarity
- no unnecessary redesign of unrelated explanation systems
- preserve current naming/style conventions in the package

Deliverable:
- stable audit-building logic aligned with the current architecture

### Stage 4. Attach audit cleanly to wmfmModel and runtime flows
Purpose: make the audit accessible wherever later rebuild stages need it.

Tasks:
- confirm `runModel()` attaches the audit to the returned object
- confirm constructors and object classes handle the field correctly
- confirm runtime/app paths can access the audit without recomputing it unnecessarily
- remove any duplicate or conflicting audit-generation paths if they exist

Likely files to inspect/change:
- `R/api-runModel.R`
- `R/class-wmfmModel.R`
- `R/app-equation-runtime.R`
- any runtime helpers that separately build explanation objects

Deliverable:
- one clear, reliable source of audit truth in the fitted model output

### Stage 5. Printing and developer inspection support
Purpose: make the audit easy to inspect during development and testing.

Tasks:
- review or rebuild `print.wmfmExplanationAudit()`
- ensure the print method is informative but stable
- make sure output is readable and deterministic
- avoid overexposing UI-specific formatting in core object methods

Deliverable:
- a useful print path for audit debugging and developer inspection

### Stage 6. Test rebuild and regression protection
Purpose: prove the audit works and stays working.

Tasks:
- add or repair unit tests for:
  - audit construction
  - audit class/shape
  - storage on `wmfmModel`
  - deterministic fields for linear and logistic examples where relevant
  - expected handling of numeric anchors / reference levels / CI information
- avoid tests that depend on live LLM calls
- update fixtures/helpers only where necessary

Likely tests:
- `tests/testthat/test-buildModelExplanationAudit.R`
- `tests/testthat/test-explanation-audit-runtime.R`
- any `runModel()` tests that should assert audit presence or structure

Deliverable:
- passing deterministic tests that guard the rebuilt audit layer

### Stage 7. Documentation and handoff for later rebuild streams
Purpose: prepare for the teaching-summary and model-explanation-panel rebuilds.

Tasks:
- update roxygen where needed
- ensure exported/internal docs match the rebuilt behaviour
- write a short handoff note describing what later stages can safely assume about the audit object
- note any fields intentionally deferred to later streams

Deliverable:
- audit subsystem ready to support later rebuild work

## Suggested order of execution
1. complete the audit inventory and gap analysis first
2. freeze the audit contract before major edits
3. rebuild core audit generation
4. wire it into `runModel()` and object construction
5. repair print/helpers only after the core object is stable
6. finish with deterministic tests and documentation

## Key risks
- rebuilding from memory rather than from code evidence
- reintroducing deleted or obsolete branch structures
- duplicating audit generation in multiple places
- mixing developer audit needs with student-facing UI needs too early
- tests passing only because they mirror stale implementation details

## Practical git workflow
Recommended workflow:

- keep working on `exp-to-exp-iv`
- do **not** merge `explaining-explanations` wholesale
- use the old branch only as a reference source during comparison
- commit the audit rebuild separately from later teaching-summary and UI rebuild work

## Suggested checkpoints before code changes begin
Before editing code, confirm all of the following:

- what the current audit code already does
- what the old branch intended the audit to do
- what exact gaps remain
- whether any current tests already cover parts of the desired behaviour
- whether any audit-related UI helpers should be ignored until later stages

## End state for this context
At the end of this rebuild stream, WMFM should have a trustworthy deterministic explanation-audit layer that is available from the fitted model object and ready to support later student-facing explanation features.
